# JDK8→JDK11更改後にG1GCのRemark Pauseが1.4秒まで伸びたのでHeapDumpで原因を特定した話

既存のJavaシステムの更改で\
**JDK8 → JDK11** のバージョンアップを実施しました。

今回の更改は次のような前提でした。

-   フレームワーク変更なし
-   基本的にストコン（現状踏襲）
-   GC設定も大きな変更なし

そのため

「JVMアップグレードによる性能問題は出ないだろう」

という前提でリリースしました。

しかし、リリース直前にある問題が発生しました。

------------------------------------------------------------------------

# 発生した問題

負荷テストにて **周期的なレスポンス遅延** が発生しました。

調査の結果、原因は **STW（Stop The World）スパイク**でした。

GCログを見ると次のログが出ていました。

    Pause Remark 1400ms

通常は

    300〜400ms

程度だったため、明らかに異常でした。

環境は次の通りです。

  項目 | 内容
  ------ ------
  - JDK    11
  - GC     G1GC
  - Heap   10GB

------------------------------------------------------------------------

# まず疑ったこと

STWが長くなる場合、まず疑うのは次の3つです。

-   Live Object増加
-   GC設定
-   メモリリーク

今回は **JDKアップグレード直後**だったため

-   G1GCの挙動
-   GC設定

も疑いました。

しかしGCログを見る限り\
GCサイクル自体は正常に回っています。

そこで次に確認したのが **HeapDump** です。

------------------------------------------------------------------------

# HeapDump取得

ヒープダンプは次のコマンドで取得しました。

``` bash
jcmd <pid> GC.heap_dump heap.hprof
```

取得したHeapDumpを解析します。

------------------------------------------------------------------------

# HeapDump解析

HeapDumpを解析すると\
あるオブジェクトが **明らかに増え続けている**ことが分かりました。

    HttpSession

Dominator Treeを見ると次の構造になっています。

    HttpSession
     └ userId (String)

この `HttpSession` が大量に残っていました。

------------------------------------------------------------------------

# 原因

コードを確認すると\
セッションに `userId` を保存していました。

``` java
session.setAttribute("userId", userId);
```

この実装自体は割と一般的です。

しかし今回のシステムでは

    セッションの解放が適切に行われていない

状態でした。

結果として

    HttpSession
     ↓
    userId
     ↓
    参照チェーン

が残り続け

    Live Object が増加

していました。（所謂セッションリーク状態）

------------------------------------------------------------------------

# なぜRemarkが遅くなるのか

G1GCのサイクルは次のようになっています。

    Initial Mark (STW)
    ↓
    Concurrent Mark
    ↓
    Remark (STW)
    ↓
    Cleanup
    ↓
    Mixed GC

この **Remarkフェーズ**では

    Concurrent Mark中に変更された参照の確定

を行います。

この処理量は **Live Object数に強く影響**されます。

今回のケースでは

    HttpSession増加
    ↓
    Live Object増加
    ↓
    Remark処理量増加
    ↓
    STWスパイク

という流れでした。

------------------------------------------------------------------------

# 修正

リリース直前ということもあり、
止血対応としてリクエスト終了時にセッション情報を削除するよう修正しました。

``` java
finally {
    session.removeAttribute("userId");
}
```

------------------------------------------------------------------------

# 修正結果

GCログは次のように改善しました。

    Pause Remark
    1400ms → 400ms

ヒープ使用量も安定しました。

------------------------------------------------------------------------

# なぜJDK8では問題が出なかったのか

今回の問題は

    JDK8 → JDK11

のアップグレードで顕在化しました。

JDK9以降では **G1GCがデフォルトGC**になっています。

参考\
https://openjdk.org/jeps/248

G1GCは

    Live Objectの量

に影響されやすいため

    参照設計の問題

がパフォーマンス問題として表面化するケースがあります。

------------------------------------------------------------------------

# 今回の学び

今回の調査で得られたポイントです。

-   GC問題は必ずしもJVM設定が原因ではない
-   HeapDumpはメモリ問題の調査に非常に有効
-   セッション設計はLive Object増加の原因になる

------------------------------------------------------------------------

# まとめ

JVMの性能問題は

    JVMチューニング

に目が行きがちですが

実際には

    アプリケーションの参照設計

が原因になっていることも多いです。

もし

    STWが突然長くなった

というケースがあれば

**HeapDumpで参照構造を確認する**

というアプローチをおすすめします。

## はじめに

「あのPR、なんかレビュー止まってない？」

大規模チームで開発していると、こういう状況が地味に発生しがちです。レビュアーも多忙で、担当者も忙しく、気づいたらPRが数日間放置されているというのは珍しくありません。

この課題を解消するために、**GitHub Actions × Microsoft Teams Webhook** を使って「PR滞留通知システム」を構築しました。

単なる通知スクリプトではなく、**複数案件への展開を見据えた3ファイル責務分離設計**を採用したので、その設計思想も含めてご紹介します。

---

## 作ったもの

- 平日午前9時に、案件ごとの Teams チャンネルへ自動通知
- 通知対象：オープンかつ最終更新から一定日数以上経過したPR（レビュアーが存在するもの）
- botによるPR（dependabotなど）は除外
- **通知内容**：リポジトリ名 / PR名 / レビュアー名 / PRリンク / リポジトリリンク
- 検知日数・対象ラベルは**案件ごとに設定可能**

---

## 設計の肝：3ファイル責務分離

最大のポイントはここです。1ファイルで全部書かず、**以下の3つのYAMLに責務を分けました**。

```
.github/workflows/
├── pr-notify-detect.yml      # 検知部：滞留PRの特定ロジック
├── pr-notify-send.yml        # 送信部：Teams への通知処理
└── pr-notify-config-xxx.yml  # 設定部：案件ごとのブランチ/Webhook設定
```

なぜ分けたのか。**新しい案件に展開するとき、設定ファイルを1つ追加するだけで完結させるためです。**

検知ロジックも送信ロジックも触らずに済む。これが大事です。

---

## 各ファイルの実装

### 1. 案件設定ファイル（`pr-notify-config-xxx.yml`）

案件ごとに1ファイル作成します。これが「案件の定義」です。

```yaml
# pr-notify-config-project-a.yml
name: PR Notify Config - Project A

on:
  workflow_dispatch:
    inputs:
      target_branch:
        description: "通知対象のマージ先ブランチ"
        default: "develop/project-a"

jobs:
  trigger-detect:
    uses: ./.github/workflows/pr-notify-detect.yml
    with:
      target_branch: "develop/project-a"
      project_name: "プロジェクトA"
      threshold_days: 1          # 何日更新がなければ検知するか
      target_label: ""           # 空白 = ラベル絞り込みなし
    secrets:
      TEAMS_WEBHOOK_URL: ${{ secrets.TEAMS_WEBHOOK_PROJECT_A }}
```

**ポイント：**
- `target_branch` でマージ先ブランチを指定 → そのブランチに向いているPRだけが対象になる
- `threshold_days` で案件ごとに検知感度を調整できる
- Webhook URLは `Secrets` に登録し、設定ファイル内には直接書かない

### 2. 検知部（`pr-notify-detect.yml`）

GitHub API で対象PRを特定します。

```yaml
# pr-notify-detect.yml
name: PR Detect (Reusable)

on:
  workflow_call:
    inputs:
      target_branch:
        required: true
        type: string
      project_name:
        required: true
        type: string
      threshold_days:
        required: false
        type: number
        default: 1
      target_label:
        required: false
        type: string
        default: ""
    secrets:
      TEAMS_WEBHOOK_URL:
        required: true

jobs:
  detect:
    runs-on: ubuntu-latest
    steps:
      - name: Detect stale PRs
        id: detect
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          TARGET_BRANCH: ${{ inputs.target_branch }}
          THRESHOLD_DAYS: ${{ inputs.threshold_days }}
          TARGET_LABEL: ${{ inputs.target_label }}
        run: |
          THRESHOLD_TIME=$(date -d "-${THRESHOLD_DAYS} days" --utc +%Y-%m-%dT%H:%M:%SZ)

          # PR一覧取得（オープン状態のみ）
          PRS=$(gh pr list \
            --repo ${{ github.repository }} \
            --base "$TARGET_BRANCH" \
            --state open \
            --json number,title,url,updatedAt,reviewRequests,labels,author \
            --limit 100)

          # 滞留PR抽出
          # ・最終更新がしきい値以前
          # ・レビュアーが1人以上いる
          # ・botは除外（author.login に [bot] を含まない）
          # ・ラベル指定がある場合はラベルフィルタ
          STALE_PRS=$(echo "$PRS" | jq --arg threshold "$THRESHOLD_TIME" \
            --arg label "$TARGET_LABEL" \
            '[.[] |
              select(.updatedAt < $threshold) |
              select(.reviewRequests | length > 0) |
              select(.author.login | test("\\[bot\\]") | not) |
              select($label == "" or (.labels[].name == $label))
            ]')

          echo "stale_prs=$(echo $STALE_PRS | jq -c .)" >> $GITHUB_OUTPUT
          echo "count=$(echo $STALE_PRS | jq length)" >> $GITHUB_OUTPUT

      - name: Call send workflow
        if: steps.detect.outputs.count != '0'
        uses: ./.github/workflows/pr-notify-send.yml
        with:
          project_name: ${{ inputs.project_name }}
          stale_prs: ${{ steps.detect.outputs.stale_prs }}
        secrets:
          TEAMS_WEBHOOK_URL: ${{ secrets.TEAMS_WEBHOOK_URL }}
```

**判定ロジックのポイント：**

| 条件 | 理由 |
|---|---|
| `state: open` | クローズ済みは対象外 |
| `updatedAt < threshold` | 最終更新日時で判定（コメント・コミット含む） |
| `reviewRequests.length > 0` | レビュアーがいないPRは除外（まだレビュー依頼前） |
| `author.login` に `[bot]` を含まない | dependabotなどの自動PRを除外 |
| ラベルフィルタ | 案件によっては特定ラベルのみ対象にしたい場合に使用 |

### 3. 送信部（`pr-notify-send.yml`）

Teams への通知を担います。

```yaml
# pr-notify-send.yml
name: PR Notify Send (Reusable)

on:
  workflow_call:
    inputs:
      project_name:
        required: true
        type: string
      stale_prs:
        required: true
        type: string
    secrets:
      TEAMS_WEBHOOK_URL:
        required: true

jobs:
  send:
    runs-on: ubuntu-latest
    steps:
      - name: Send Teams notification
        env:
          WEBHOOK_URL: ${{ secrets.TEAMS_WEBHOOK_URL }}
          PROJECT_NAME: ${{ inputs.project_name }}
          STALE_PRS: ${{ inputs.stale_prs }}
        run: |
          # PR一覧をMarkdown形式で整形
          BODY=$(echo "$STALE_PRS" | jq -r '.[] |
            "**[" + .title + "](" + .url + ")**\n" +
            "- リポジトリ: [" + (.url | split("/")[4]) + "](" + (.url | split("/")[0:5] | join("/")) + ")\n" +
            "- レビュアー: " + ([.reviewRequests[].login] | join(", ")) + "\n"
          ')

          # Teams Adaptive Card 形式で送信
          PAYLOAD=$(jq -n \
            --arg project "$PROJECT_NAME" \
            --arg body "$BODY" \
            '{
              "type": "message",
              "attachments": [{
                "contentType": "application/vnd.microsoft.card.adaptive",
                "content": {
                  "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
                  "type": "AdaptiveCard",
                  "version": "1.4",
                  "body": [
                    {
                      "type": "TextBlock",
                      "text": ("🔔 【" + $project + "】レビュー滞留PR通知"),
                      "weight": "Bolder",
                      "size": "Medium"
                    },
                    {
                      "type": "TextBlock",
                      "text": $body,
                      "wrap": true
                    }
                  ]
                }
              }]
            }')

          curl -s -X POST \
            -H "Content-Type: application/json" \
            -d "$PAYLOAD" \
            "$WEBHOOK_URL"
```

### 4. スケジューラー（`pr-notify-schedule.yml`）

平日午前9時（JST）に全案件の設定ファイルをキックします。

```yaml
# pr-notify-schedule.yml
name: PR Notify Scheduler

on:
  schedule:
    - cron: "0 0 * * 1-5"  # UTC 00:00 = JST 09:00、平日のみ

jobs:
  trigger-all:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        config:
          - workflow: pr-notify-config-project-a.yml
          - workflow: pr-notify-config-project-b.yml
    steps:
      - name: Trigger config workflow
        uses: actions/github-script@v7
        with:
          script: |
            await github.rest.actions.createWorkflowDispatch({
              owner: context.repo.owner,
              repo: context.repo.repo,
              workflow_id: '${{ matrix.config.workflow }}',
              ref: 'main'
            });
```

## Secrets の設定

Webhook URLは絶対にYAMLに直書きしないこと。`Settings > Secrets and variables > Actions` で案件ごとに登録します。

```
TEAMS_WEBHOOK_PROJECT_A  →  https://xxx.webhook.office.com/webhookb2/...
TEAMS_WEBHOOK_PROJECT_B  →  https://xxx.webhook.office.com/webhookb2/...
```

設定ファイルからは `${{ secrets.TEAMS_WEBHOOK_PROJECT_A }}` で参照するだけです。

## 通知イメージ

Teamsに届く通知はこんな感じです：

```
🔔 【プロジェクトA】レビュー滞留PR通知

[feat: ユーザー認証機能の追加](https://github.com/.../pull/123)
- リポジトリ: my-repo
- レビュアー: tanaka, yamada

[fix: ログイン画面のバリデーション修正](https://github.com/.../pull/118)
- リポジトリ: my-repo
- レビュアー: sato
```

リンクをクリックすればそのままPRに飛べるので、「通知が来た → 即アクション」ができます。

## 新しい案件への展開方法

この設計の真価がここです。**新しいプロジェクトCに展開したい場合、やることは2つだけ。**

1. `pr-notify-config-project-c.yml` を作成（既存ファイルをコピーして、ブランチ名と案件名を変更）
2. `TEAMS_WEBHOOK_PROJECT_C` をSecretsに追加
3. スケジューラーのmatrixに1行追加

検知ロジックも送信ロジックも一切触らない。これが責務分離の恩恵です。

## 実際に運用してみて

4案件で運用を開始しました。

**よかった点：**
- 「あのPRどうなってる？」という確認コストが激減
- 通知が来ることで、レビュアー側も「早めに対応しなきゃ」という意識が生まれた
- 案件ごとに通知チャンネルが分かれているので、ノイズにならない

**チューニングポイント：**
- 最初は `threshold_days: 1` で設定したが、チームによっては `2` にしてもよさそう
- レビュアーが多い案件は滞留が少なく、通知頻度が低めになる傾向

## まとめ

| 要素 | 内容 |
|---|---|
| トリガー | 平日午前9時（cron） |
| 検知条件 | オープン / 最終更新から指定日数以上 / レビュアーあり / botは除外 |
| 設定単位 | 案件ごとにyml1ファイル |
| 通知先 | 案件ごとのTeamsチャンネル（Webhook） |
| 機密管理 | Webhook URLはSecretsで管理 |

「PR滞留を検知する仕組みは欲しいけど、案件が増えるたびに設定が複雑になるのは困る」という課題に対して、**責務を明確に分離した3ファイル構成**が有効でした。

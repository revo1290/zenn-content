## はじめに

最近よく「ハーネス」って言葉を目にするようになりました。エージェントを動かす足場、みたいな意味で使われているやつです。

正直、自分にとっては雰囲気でしかわかってない概念でした。でも、なんか面白そうやなと思って、**素人知識で見よう見まねで組んでみた**んです。

そしたら、**1リクエスト投げるだけで最大4スプリントぶん自律的に回る**ところまで来てしまいました。業務の一部タスクで実用してるレベルでは動いてます。

> 「あれ、ハーネスってもしかしてこういうこと？」

っていう自分なりの気づきを、実験メモ的に残しておきます。同じく雰囲気で理解してる人の参考になれば嬉しいです。

:::message
本記事は「完成された設計」の紹介ではなく、**試行錯誤の過程**を残すものです。なので「こうするのが正解！」ではなく「こうしたら回りました」レベルの話として読んでもらえたら。
:::

## 最初に作ろうとしたもの

最初のモチベーションはシンプルで、「エージェントを複数用意してスクラムっぽく回せへんかな？」でした。

基盤は GitHub Copilot。`.agent.md` でエージェントを定義できるので、以下の役割で用意してみました。

- 実装スペシャリスト
- バックエンドスペシャリスト
- デザインスペシャリスト（フロントエンド実装もここ）
- コンテナスペシャリスト
- テストスペシャリスト

オーケストレーターから各スペシャリストを呼ぶ、よくある構成です。

## 最初の壁：テスト実行がちょいちょい漏れる

動かしてみたら、それっぽくコードは出てくるんですが、**テスト実行が漏れる**ことがちょいちょいあるんです。

「テストスペシャリスト」に任せているはずなのに、気づいたらテストを書いただけで終わってる。実行してない。

原因を考えてみたら、**設計と実行を同じエージェントに任せていた**のが問題でした。1つのエージェントに2つの責務を持たせると、どっちかが必ず疎かになります。

### 解決：責務を分離した

- テストスペシャリスト → **テスト設計専任**
- テスト実行エージェント → **既存テスト全件（リグレッション含む）の実行専任**

これだけでテスト漏れが劇的に減りました。

> 気づき1：**1エージェントに複数責務を持たせない**

## 次の壁：品質が安定しない

テストは実行されるようになったんですが、今度は「動いてるけど要件を満たしてない」実装が出てくる問題に直面しました。

実装エージェント自身が「できました！」って言ってるのに、よく見たら受入条件を半分しか満たしてない、みたいな。

これを解決するために、**QAチーム**というガードレールを設けることにしました。

### QAチームを3つの役割に分けた

最初は「QAエージェント」1つで作ってたんですが、これまた同じ問題で、チェック項目が漏れる。なので分離。

|担当       |責務                       |
|---------|-------------------------|
|静的解析担当   |lint / type check / コード品質|
|要件照合担当   |実装が受入条件を満たしているかコードベースで検証 |
|テスト結果検証担当|テスト実行結果の検証               |

判定ルールはシンプルにしました。**3つ全員が ACCEPT したらスプリント完了**。1つでも REJECT なら差し戻し。

> 気づき2：**ガードレールは実行役から独立させる**。自己チェックは信用ならない。

## 3つ目の壁：自己改善ループが最初の方しか回らない

ここから本題です。**ハーネスっぽさ**の話。

KPT（Keep / Problem / Try）でスプリントごとに振り返って、失敗談をナレッジに残す仕組みを組みました。各エージェントが自分の結果を見て自己内省して KPT を出す形です。

最初は回ってました。ちゃんと Problem と Try がナレッジに溜まっていって、次スプリントで参照される感じ。

**でもスプリントを重ねるうちに、振り返りはするけどナレッジに残らなくなった**んです。

### 原因：オーケストレーターが忙しすぎた

よく見ると、オーケストレーターが「タスク分配」と「フェーズ管理」の両方を担っていて、負荷が高いとフェーズ管理（振り返り→ナレッジ反映）の方が抜けていました。

### 解決：Hand-off エージェントを新設

振り返りを**必須ステップ**として `.agent.md` に徹底させたうえで、オーケストレーターとは別に **Hand-off エージェント**を作りました。

- 動くタイミング：スプリント開始時と完了時のみ
- 責務：フェーズ間の引き継ぎ情報をパッケージ化する

つまり、「スプリントが終わった」「次のスプリントが始まる」というイベントだけを確実に管理する専任役です。

この分離で、ナレッジ反映が確実に実行されるようになりました。

> 気づき3：**イベント管理は独立したエージェントに任せる**。オーケストレーターに全部持たせるとスケールしない。

## できあがった構成

試行錯誤の結果、今はこんな感じになっています。

```
                   ┌──────────┐
                   │  User    │ ← バックログ投入 / 重要変更承認
                   │  (PO)    │   受入条件承認 / エスカレーション対応
                   └─────┬────┘
                         │
            ┌────────────▼────────────┐
            │     Orchestrator         │  ← タスク分配 / SP算出
            │  (ルール＋LLMハイブリッド)│    スプリント計画
            └────┬─────────────────┬───┘
                 │                 │
     ┌───────────▼──┐      ┌───────▼────────────┐
     │ Hand-off     │      │  docs/             │
     │  Agent       │      │   ├─ backlog.md    │
     │ (フェーズ境界 │      │   ├─ velocity.md   │
     │  情報継承)   │      │   └─ lesson_known.md│
     └──────────────┘      └────────────────────┘
                 │
     ┌───────────▼───────────────────────────────┐
     │         Specialist Agents (並列実行可)    │
     │ 実装 / BE / デザイン(FE含) / コンテナ /    │
     │ テスト設計 / テスト実行                    │
     └───────────┬───────────────────────────────┘
                 │
     ┌───────────▼─────────────────────┐
     │         QA Team                  │
     │ (全員ACCEPT必須)                 │
     │  ├─ 静的解析担当                 │
     │  ├─ 要件照合担当                 │
     │  └─ テスト結果検証担当           │
     └──────────────────────────────────┘
```

### ドキュメント管理

`docs/` 配下に3つのMDファイルを置いて、状態を永続化しています。

|ファイル                  |役割                   |
|----------------------|---------------------|
|`docs/backlog.md`     |プロダクトバックログ・スプリントバックログ|
|`docs/velocity.md`    |スプリントごとの実績・ベロシティ・所要時間|
|`docs/lesson_known.md`|ナレッジ・失敗談・設計方針        |

全エージェントが作業開始時にこれらを読み込む、という指示を徹底しています。

### スプリント計画の仕組み

- ストーリーポイントはオーケストレーターが**タスク内容＋過去実績**から算出
- 1スプリント上限13ポイント、超えたら自動でスプリント分割
- ベロシティは実績ベースで積み上げ、各エージェントの完了時間も記録

### ガードレール

- 重要変更（DB スキーマ、ライブラリ選定、外部APIの破壊的変更）は**人間の承認必須**
- スプリント内で N 回 REJECT されたら中断して**人間にエスカレーション**
- バックログはユーザーが PO として投入、受入条件はオーケストレーター生成→ユーザー承認

## 「ハーネスってこういうこと？」

ここまで作ってみて、自分なりに腑に落ちた感覚があります。

**ハーネスって、エージェントに「何をするか」を書くプロンプトとは別に、エージェントが「どう協調して動くか」を定める足場のことなんちゃうか**、と。

具体的には：

1. **状態の永続化** — `docs/` 配下の3ファイルがまさにこれ。エージェントは都度起動されて記憶を持たないので、状態はファイルで管理する
1. **責務の明確な分離** — 1エージェントに1責務。設計と実行、実行とガードレール、フェーズ管理とタスク分配を分ける
1. **イベント駆動のフェーズ管理** — スプリント境界のような「区切り」を専任エージェントに任せる
1. **自己改善ループの強制** — 振り返り→ナレッジ反映を仕組みとして組み込む

プロンプトエンジニアリングが「個のエージェントを賢くする」話なら、ハーネスは「**エージェントたちが協調して失敗しないようにする**」話なんやな、と。

もちろん、これが「正解のハーネス」とは思ってません。ただ、**素人がそれっぽく組んでみたら、試行錯誤の結果これに落ち着いた**というのが正直なところです。

## 実際どのくらい動いてる？

定量的には：

- 1リクエストで**最大4スプリント**自律実行
- 1スプリントあたり**数分〜10分**
- スプリント重ねるごとに QA リジェクト率は下がる（ナレッジが溜まる効果）

定性的には：

- 実装のためのコーディング時間が圧倒的に減った
- 品質が安定した（QA ガードレール効果が大きい）
- ドキュメントや振り返りが自動で生成される
- 自分は設計・要件定義・重要判断に集中できるようになった

## まとめ

素人なりにハーネスというものに挑んでみたら：

- **責務を分けないと必ずどこかが漏れる**
- **ガードレールは実行役から独立させないと機能しない**
- **フェーズ境界のイベント管理は専任エージェントに任せると安定する**
- **状態は MD ファイルで永続化するだけでも、意外と回る**

という学びがありました。

プロンプトだけじゃなくて、**エージェントをどう並べて、どう会話させて、どう状態を持たせるか**が重要なんやなと。それがハーネスなんやろう、というのが今の自分の理解です。

まだまだ雰囲気でやってる部分も多いので、「こうした方がええよ」っていうツッコミもらえたら嬉しいです。
